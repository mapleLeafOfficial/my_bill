# 项目模块化开发流程 (Modular Development Workflow)

## 概述

本 Skill 定义了一套标准化的 AI 辅助项目开发流程，核心理念是：**分阶段推进、逐模块开发、每步确认**。

整体流程：
```
用户输入需求 → [阶段一] 生成 PRD.md → 用户确认
    → [阶段二] 拆分为 todo1~N.md → 用户确认
    → [阶段三] 开发 todo1 → 用户确认 → 开发 todo2 → ...
```

---

## 阶段零：项目上下文调研（必须首先执行）

在开始任何工作之前，**必须**完成以下调研：

### 步骤 0.1：读取项目规范文件

按顺序查找并阅读以下文件（如果存在）：
1. `CLAUDE.md`（项目根目录）
2. `AGENTS.md`（项目根目录）
3. `项目初始化.md`（项目根目录）
4. `.cursor/rules` 或其他规范文件

> **关键信息提取**：
> - 技术栈（前端/后端框架、语言版本）
> - 构建命令（启动、测试、SDK 生成）
> - 代码规范（架构风格、命名约定）
> - **特殊开发流程**（例如：后端修改后需要重启并运行 `npm run openapi`）

### 步骤 0.2：了解项目结构

```
列出项目根目录的文件和目录结构（排除 node_modules、.git、target 等）
```

### 步骤 0.3：检查已有文档

检查 `/doc/` 目录是否已存在 PRD 或计划文件，避免重复工作。

---

## 阶段一：生成产品需求文档 (PRD)

### 输入

用户提供的需求描述，可以是：
- 自然语言描述
- 功能列表
- 参考文档或截图
- 现有的需求文件

### 执行步骤

1. **收集需求**：如果用户提供的信息不够明确，主动询问补充信息：
   - 核心功能有哪些？
   - 目标用户是谁？
   - 有没有可参考的系统？
   - 有没有非功能性需求（性能、安全等）？

2. **生成 PRD 文档**：在项目根目录下创建 `/doc/PRD.md`，参考模板 `templates/prd-template.md`。
   - PRD 必须包含：项目概述、功能模块列表、每个模块的详细描述、数据模型设计、非功能性需求。
   - 功能模块应按优先级排序。
   - 每个功能模块应明确标注其涉及的前后端组件。

3. **⛔ 停下来**：使用 `notify_user` 工具，将 PRD 路径放入 `PathsToReview`，请用户审阅。
   - 设置 `BlockedOnUser = true`
   - 消息示例：「PRD 已生成，请审阅并确认。如有修改意见请直接说明。确认后我将进行模块拆分。」

### 输出

- `/doc/PRD.md` — 完整的产品需求文档

---

## 阶段二：模块化计划拆分

### 前置条件

用户已确认 PRD。

### 执行步骤

1. **阅读 PRD**：重新阅读 `/doc/PRD.md` 获取最新内容（用户可能已修改）。

2. **拆分模块**：将 PRD 中的功能模块拆分为可独立开发的 todo 文件。

   拆分原则：
   - **一个 todo = 一个可独立运行的功能模块**
   - **按依赖顺序排列**：基础模块（如数据模型、认证）排在前面
   - **粒度适中**：每个 todo 应在一个对话轮次内能完成（约 15-30 个文件变更）
   - **前后端联动**：如果一个功能同时涉及前后端，将它们放在同一个 todo 中

3. **生成 todo 文件**：在 `/doc/plan/` 目录下创建 `todo1.md`、`todo2.md`、...、`todoN.md`。
   - 参考模板 `templates/todo-template.md`
   - 每个 todo 文件必须包含：模块名称、目标描述、具体任务清单、涉及的文件列表、依赖关系、验证标准

4. **生成总览文件**：创建 `/doc/plan/README.md`，汇总所有 todo 的概览和当前进度。

5. **⛔ 停下来**：使用 `notify_user` 工具通知用户审阅。
   - 将 `/doc/plan/README.md` 放入 `PathsToReview`
   - 消息示例：「已将 PRD 拆分为 N 个开发模块，请审阅计划总览。确认后我将从 todo1 开始开发。」

### 输出

- `/doc/plan/README.md` — 计划总览
- `/doc/plan/todo1.md` ~ `/doc/plan/todoN.md` — 各模块计划

## 阶段三：逐模块开发

### 前置条件

用户已确认计划拆分。

### 核心原则

> **每次只开发一个 todo 模块。** 
> 完成后必须停下来，等待用户确认再继续下一个。
> 这是为了保持上下文的干净和可控。

### 执行步骤（每个 todo 循环执行）

#### 步骤 3.1：开始一个 todo

1. 阅读当前 todo 文件（例如 `/doc/plan/todo1.md`）
2. 重新阅读项目规范文件（`CLAUDE.md` / `AGENTS.md`）以确保遵循规范
3. 使用 `task_boundary` 工具设置当前任务

#### 步骤 3.2：执行开发

按照项目规范执行开发，通用开发顺序：

**后端优先项目（如 Spring Boot + Vue）：**

```
1. 创建/修改数据模型（Entity、DTO）
2. 创建/修改 Repository
3. 创建/修改 Service
4. 创建/修改 Controller（含完整 Swagger 注解）
5. 运行后端测试（如果有）: mvn test
6. 【关键】重启后端服务以更新 OpenAPI 文档
7. 【关键】在 frontend 目录运行 npm run openapi 生成前端 SDK
8. 创建/修改前端页面组件
9. 创建/修改前端路由
10. 创建/修改前端状态管理（如需要）
```

**纯前端项目：**
```
1. 创建/修改组件
2. 创建/修改路由
3. 创建/修改状态管理
4. 创建/修改样式
```

**注意事项一：读取项目特定流程**

每个项目的开发流程可能不同，务必从阶段零中提取的项目规范里确认：
- 后端修改后是否需要重启？
- 是否需要运行 SDK 生成命令？（如 `npm run openapi`）
- 是否有特定的代码生成工具？
- 测试命令是什么？

**注意事项二：遵循项目编码规范**

- 使用项目规定的依赖注入方式
- 使用项目规定的响应结构
- 使用项目规定的错误处理方式
- 使用项目规定的命名规范
- 禁止手动编写自动生成的文件（如前端 API SDK）

#### 步骤 3.3：验证

1. 确保代码编译/构建通过
2. 运行项目指定的测试命令
3. 如果涉及后端 API 变更且项目要求，执行 SDK 重新生成

#### 步骤 3.4：更新进度

1. 在当前 todo 文件中将已完成的任务标记为 `[x]`
2. 更新 `/doc/plan/README.md` 中的进度状态

#### 步骤 3.5：⛔ 停下来

使用 `notify_user` 工具通知用户：

```
消息模板：
「todo{N}（{模块名称}）已完成开发。

完成的内容：
- xxx
- xxx

下一个待开发模块：todo{N+1}（{模块名称}）

是否继续开发下一个模块？」
```

- 设置 `BlockedOnUser = true`
- 设置 `ShouldAutoProceed = false`（必须等用户确认）

### 当所有 todo 完成后

使用 `notify_user` 通知用户全部开发完成：
```
消息模板：
「所有 {N} 个模块已全部完成开发。

完成模块列表：
1. todo1 - {模块名} ✅
2. todo2 - {模块名} ✅
...

请进行整体测试验收。」
```

---

## 重要规则

### 1. 每阶段必须确认

**严禁跳过用户确认环节。** 每个阶段结束必须使用 `notify_user` 通知用户并等待确认。

### 2. 每次只做一个 todo

**严禁在一个对话中开发多个 todo。** 完成一个后必须停下来等待确认。

### 3. 遵守项目规范

**读取并严格遵守项目的 `CLAUDE.md` / `AGENTS.md` 中的规范。** 特别注意：
- 禁止手动修改自动生成的文件
- 后端接口变更后必须执行 SDK 重新生成流程
- 使用项目规定的架构层级和编码风格

### 4. 保持文档同步

每个 todo 完成后，**必须更新** `/doc/plan/README.md` 中的进度。

### 5. 中文交流

所有思考过程、任务说明、代码注释以外的文档，均使用中文。

---

## 错误恢复

### 如果 todo 开发中途失败

1. 记录失败原因到 todo 文件的备注区域
2. 将未完成的任务保持 `[ ]` 状态
3. 通知用户失败原因，询问是否需要调整方案

### 如果用户要求修改已完成的 todo

1. 重新阅读对应的 todo 文件
2. 根据用户反馈执行修改
3. 重新验证
4. 更新 todo 状态

### 如果需要调整计划

1. 更新 `/doc/plan/README.md`
2. 创建新的 todo 文件或修改现有的
3. 通知用户审阅修改后的计划
